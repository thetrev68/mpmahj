# Archive

This directory contains **completed work artifacts** that provide historical context but are no longer active reference material.

## Directory Structure

### `plans/`

Implementation planning documents that have been **executed and completed**. These documents describe the approach taken before implementation began.

**When to archive plans:**

- The implementation is complete
- A corresponding implementation summary exists
- The code is in the codebase and working

**Do NOT archive:**

- Plans for in-progress work
- Plans for future work

### `summaries/`

Implementation completion reports that document **what was built and how**. These are retrospective documents created after implementation.

**When to archive summaries:**

- After a stabilization period (e.g., 1-2 weeks post-implementation)
- Once the implementation details are well-understood by the team
- When the active documentation has been updated to reflect the implementation

**Do NOT archive:**

- Recent implementation summaries (keep accessible for reference)
- Summaries with unresolved TODOs or known issues

## When to Reference Archive Files

Archive files are useful for:

- Understanding **why** decisions were made
- Reviewing the **original scope** of a feature
- Onboarding new team members who need historical context
- Debugging issues related to implementation choices

## Related Directories

- `docs/architecture/` - Active reference documentation (system design, patterns)
- `docs/implementation/` - Current implementation documentation
- `docs/deprecated/` - Obsolete documentation that has been superseded
